PR(Pull Request) 리뷰어 관점에서 변경 사항을 검토합니다.

## 리뷰 체크리스트

### 기능 검토
- [ ] 의도한 기능이 올바르게 구현됐는가
- [ ] 엣지 케이스가 처리됐는가
- [ ] 기존 기능에 영향을 주지 않는가

### 코드 품질
- [ ] 변수명/함수명이 의미 있고 명확한가
- [ ] 중복 코드가 없는가
- [ ] 복잡한 로직에 주석이 있는가

### ROS2 특화 검토
- [ ] QoS(Quality of Service) 설정이 publisher/subscriber 간 호환되는가
- [ ] launch 파라미터가 실제 노드 파라미터와 일치하는가
- [ ] 절대 경로 사용하지 않았는가
- [ ] UI 파일과 알고리즘 파일이 분리되어 있는가
- [ ] CMakeLists.txt / package.xml 올바른가
- [ ] colcon build 성공하는가

### TR-Works 규칙
- [ ] 커밋 메시지 접두어 규칙 준수 (feat/fix/docs...)
- [ ] 불필요한 파일 포함 여부 (.env, 개인 설정 등)
- [ ] config 파일 형식 올바른가 (YAML 문법)

### 보안
- [ ] 민감 정보(API 키, 비밀번호)가 포함됐는가
- [ ] 하드코딩된 경로/URL이 있는가

## 실행

1. `git diff origin/main...HEAD` 로 전체 변경 내용 확인
2. 위 체크리스트 기준으로 각 항목 평가
3. 발견된 이슈를 심각도로 분류해서 리포트:
   - 필수 수정: 빌드 실패, QoS 불일치, 보안 문제
   - 권장: 코드 품질, 네이밍, 중복
   - 제안: 개선 아이디어, 더 나은 방법
4. 잘 작성된 부분도 긍정 피드백 포함
